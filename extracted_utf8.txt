Tema 2. Programación en SQL.
La programación en SQL permite ampliar las capacidades de un sistema
de gestión de bases de datos, incorpora estructuras que van más allá de
las consultas básicas. A través de vistas, funciones, procedimientos
almacenados y triggers, es posible encapsular lógica de negocio,
automatizar tareas y optimizar el acceso a la información, estas
herramientas proporcionan mayor seguridad, reutilización de código y
eficiencia en la gestión de datos (Gupta & Ramachandra, 2021).
2.1. Vistas y Funciones.
Vistas.
Una vista es una tabla virtual que se construye a partir de una o varias
tablas reales mediante una consulta SQL, su objetivo es simplificar el
acceso a la información, ocultar la complejidad de las consultas y
mejorar la seguridad, ya que permite restringir las columnas o filas visibles
para ciertos usuarios, las vistas son muy útiles para reutilizar consultas
frecuentes, estandarizar reportes y proporcionar diferentes perspectivas
de los datos sin duplicarlos físicamente, aunque no almacenan datos por
sí mismas, pueden actualizarse si cumplen determinadas condiciones
(Silberschatz, 2021).
Ejemplo:
Código SQL
-- C rear una vista para mostrar solo pedidos activos con datos del
clie nte
CREATE VIEW Vista_PedidosActivos AS
SELECT c.nombre, p.id_pedido, p.fecha, p.monto
FROM clientes c
JOIN pedidos p ON c.cliente_id = p.cliente_id
WHER E p.estado = 'ACTIVO';
-- C onsultar la vista como si fuera una tabla
SELECT * FROM Vista_PedidosActivos;

Vistas Actualizables.
Una vista actualizable es aquella que permite realizar operaciones de
modificación de datos, como INSERT, UPDATE o DELETE, directamente
sobre ella, reflejándose los cambios en la tabla base de la que proviene,
para que esto sea posible, la vista debe construirse sobre una sola tabla
y no incluir elementos que alteren la correspondencia directa con sus filas,
como agregaciones, subconsultas o uniones (JOIN) (Silberschatz, 2021).
Este tipo de vistas facilita el acceso restringido a los datos sin perder
capacidad de edición.
Ejemplo:
Código SQL
-- Crear una vista sencilla de clientes
CREATE VIEW Vista_Clientes AS
SELE CT cliente_id, nombre, ciudad
FROM clientes;
-- Insertar un nuevo cliente a través de la vista
INSERT INTO Vista_Clientes (cliente_id, nombre, ciudad)
VALUES (3001, 'Ana Torres', 'Quito');
-- A ctualizar la ciudad de un cliente desde la vista
UPDA TE Vista_Clientes
SET ciudad = 'Guayaquil'
WHERE cliente_id = 3001;
-- Consultar la vista
SELE CT * FROM Vista_Clientes;

2.2. Procedimientos almacenados.
Un procedimiento almacenado es un conjunto de instrucciones SQL que
se guarda en el servidor de base de datos y puede ejecutarse al
momento que sea necesario, a diferencia de las funciones, que siempre
devuelven un valor o conjunto de resultados, los procedimientos
almacenados pueden realizar múltiples operaciones: insertar, actualizar,
eliminar, validar reglas de negocio o incluso llamar a otros procedimientos
y funciones (Silberschatz, 2021).

2.3. Triggers.
Un trigger es un programa almacenado que se ejecuta
automáticamente en respuesta a eventos INSERT, UPDATE o DELETE sobre
una tabla o vista, puede ser BEFORE/INSTEAD OF (antes/sustituye la
operación) o AFTER (después), y a nivel fila o sentencia, según el DBMS.
Sirve para hacer cumplir reglas de negocio, auditar cambios, mantener
integridad derivada y sincronizar datos entre tablas, sin requerir código
en la aplicación (Connolly & Begg, 2021).
Uso adecuado de triggers en bases de datos
Los triggers deben emplearse si se necesita que ciertas acciones se
ejecuten automáticamente al ocurrir un evento en la base de datos, sin
depender de la aplicación (Silberschatz, 2021).
Son especialmente útiles en tres escenarios:
1. Auditoría de cambios: Registrar quién, cuándo y qué valores se
modificaron en una tabla (por ejemplo, cambios de estado en
un pedido).
2. Validación de reglas de negocio: Garantizar que se cumplan
condiciones que no pueden controlarse solo con restricciones
simples, como evitar saldos negativos o impedir la eliminación
de un registro crítico.
3. Mantenimiento automático de datos relacionados: Actualizar
totales, recalcular saldos o sincronizar información en tablas
dependientes cada vez que ocurre un cambio.
Sin embargo, no deben usarse para lógica muy compleja ni para
reemplazar restricciones (PRIMARY KEY, FOREIGN KEY, CHECK) que ya

2.4. Optimización de consultas, Ejercicios de Aplicación.
Optimización de consultas.
La optimización de consultas es el proceso mediante el cual un sistema
gestor de bases de datos elige la forma más eficiente de ejecutar una
instrucción SQL, una consulta mal diseñada puede consumir demasiados
recursos, tardar en responder o bloquear otras operaciones (Rahman
et al., 2024).
Entre las principales estrategias de optimización están:
1. Uso de índices adecuados: Aprovechar estructuras que
aceleran búsquedas.
2. Selección precisa de columnas y filas: Evitar SELECT *.
3. JOIN bien planteados: Usar las claves correctas y el orden lógico
de combinación.
4. Evitar subconsultas innecesarias: Reemplazarlas por JOIN o CTE
si corresponde.
5. Análisis de planes de ejecución: Revisar cómo el motor procesa
la consulta para detectar cuellos de botella.
Con estas prácticas se logra reducir el consumo de memoria y tiempo de
CPU, mejora el rendimiento global de la base de datos.

Código SQL
-- Consulta poco eficiente
SELECT *
FROM pedidos p, clientes c
WHERE p.cliente_id = c.cliente_id

-- Consulta optimizada
SELECT c.nombre, p.id_pedido, p.monto
FROM pedidos p
INNER JOIN clientes c ON p.cliente_id = c.cliente_id
WHERE p.monto > 500;
La segunda versión es más eficiente porque: Selecciona solo las
columnas necesarias; usa sintaxis explícita de JOIN; permite al
optimizador aplicar índices en monto y cliente_id.

FROM total_por_cliente
