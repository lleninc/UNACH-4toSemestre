1.1

Rol y Funciones del DBA

▼

La administración de bases de datos es la disciplina que planifica, organiza y controla los datos almacenados en un SGBD, garantizando disponibilidad, integridad, seguridad y accesibilidad.

Definición: El DBA es el profesional responsable de garantizar que la información sea segura, íntegra y esté disponible. Combina conocimientos técnicos con habilidades para resolver problemas y tomar decisiones oportunas.

Funciones principales del DBA:

  • Instalar y mantener el gestor de base de datos (SGBD)
  • Diseñar estructuras de almacenamiento y organización
  • Administrar usuarios, roles y permisos de acceso
  • Implementar seguridad, respaldos y recuperación de datos
  • Monitorear rendimiento, prevenir fallos y proponer mejoras

Buenas prácticas:

  • Políticas periódicas de respaldo y recuperación
  • Control de acceso por roles (principio de mínimo privilegio)
  • Actualización constante de parches de seguridad
  • Monitoreo continuo: CPU, memoria, I/O, tiempos de respuesta
  • Documentar procesos y probar antes de cambios críticos
  • Aplicar estándares de normalización
Clave para examen: El DBA NO diseña campañas de marketing, NO crea aplicaciones para usuarios finales, NO edita imágenes. Su perfil es 100% técnico: diseño, configuración y optimización de bases de datos.
1.2

Transacciones y Propiedades ACID

▼

Una transacción es una secuencia de operaciones que se ejecuta como una unidad indivisible. Si algún paso falla, se revierten todas las operaciones (ROLLBACK) para evitar inconsistencias.

Ejemplo bancario: Transferir $500 implica (1) debitar cuenta origen y (2) acreditar cuenta destino. Si el paso 2 falla → ROLLBACK de ambos pasos. Todo o nada.
A
Atomicidad
Todo se ejecuta o nada se ejecuta. La transacción es una unidad indivisible. Si falla → ROLLBACK completo.
C
Consistencia
Se respetan todas las reglas y restricciones de la BD. No se pueden generar estados inválidos.
I
Aislamiento
Las transacciones concurrentes no se interfieren entre sí. Los cambios intermedios no son visibles para otras transacciones.
D
Durabilidad
Los cambios confirmados (COMMIT) persisten permanentemente, incluso ante fallos del sistema.

Comandos SQL de transacción:

BEGIN TRANSACTION;

-- Paso 1: Debitar cuenta origen
UPDATE cuentas SET saldo = saldo - 500 WHERE id = 1;

-- Paso 2: Acreditar cuenta destino
UPDATE cuentas SET saldo = saldo + 500 WHERE id = 2;

COMMIT;  -- Confirmar cambios
-- Si ocurre error antes del COMMIT → ROLLBACK automático
Diferencia COMMIT vs ROLLBACK: COMMIT confirma y persiste los cambios (durabilidad). ROLLBACK cancela y deshace todos los cambios de la transacción actual.
1.3

Control de Bloqueos y Niveles de Aislamiento

▼

Un bloqueo es el mecanismo que usan los SGBD para coordinar el acceso concurrente y evitar inconsistencias como actualizaciones perdidas y estados intermedios visibles.

Tipos básicos:
🔵 Compartido (S): Permite lectura concurrente por múltiples transacciones.
🔴 Exclusivo (X): Solo una transacción puede escribir; las demás esperan hasta el COMMIT/ROLLBACK.

Granularidad de bloqueo (de mayor a menor concurrencia):

  • Fila: Máxima concurrencia, menor impacto
  • Página: Bloqueo intermedio
  • Tabla: Menor concurrencia, mayor impacto

Niveles de Aislamiento SQL (de menor a mayor seguridad):

Nivel Lectura sucia No repetible Fantasma Uso
Read Uncommitted ✗ posible ✗ posible ✗ posible Reportes no críticos
Read Committed ✓ evita ✗ posible ✗ posible Producción general
Repeatable Read ✓ evita ✓ evita ✗ posible Lecturas críticas
Serializable ✓ evita ✓ evita ✓ evita Máxima seguridad
Clave: Read Committed = mejor equilibrio rendimiento/seguridad (más usado en producción). Serializable = más estricto, equivale a ejecución secuencial.
1.4

Índices y Sentencias Avanzadas

▼

Un índice es una estructura auxiliar (generalmente un Árbol B / B-Tree) que mantiene claves ordenadas para acelerar búsquedas, JOIN, ORDER BY y GROUP BY.

Trade-off clave: Los índices mejoran la lectura (SELECT) pero añaden costo a escritura (INSERT/UPDATE/DELETE), porque cada cambio debe actualizar también los índices. Demasiados índices → rendimiento de escritura degradado.
Clustered (Agrupado)
Define el orden físico de almacenamiento de la tabla. Solo puede existir UNO por tabla. Ideal para rangos. Las PKs son clustered por defecto.
Non-Clustered (No agrupado)
Estructura separada con claves ordenadas y punteros. Pueden existir varios por tabla. Ideal para búsquedas puntuales por columnas específicas.
-- Índice agrupado (1 por tabla)
CREATE CLUSTERED INDEX IX_Pedidos_Id
ON Pedidos (id_pedido);

-- Índice no agrupado (múltiples por tabla)
CREATE NONCLUSTERED INDEX IX_Pedidos_Cliente
ON Pedidos (cliente_id);

Sentencias Avanzadas:

  • JOIN: Combina tablas (INNER, LEFT, RIGHT, FULL)
  • Subconsultas: Anidadas en WHERE/FROM/SELECT. No correlacionada (ejecuta 1 vez) vs correlacionada (ejecuta por fila)
  • CTE (WITH): Tablas temporales nombradas para organizar consultas complejas
  • Consultas recursivas: WITH RECURSIVE para jerarquías (organigramas, árboles)
  • Funciones de ventana (OVER): Cálculos por grupo sin colapsar filas individuales
-- CTE: Simplifica consultas complejas
WITH compras_cliente AS (
    SELECT cliente_id, SUM(monto) AS total
    FROM pedidos
    GROUP BY cliente_id
)
SELECT c.nombre, cc.total
FROM clientes c
JOIN compras_cliente cc ON c.cliente_id = cc.cliente_id
WHERE cc.total > 1000;

-- Función de ventana: ranking sin perder detalle
SELECT cliente_id, SUM(monto) AS total,
       RANK() OVER (ORDER BY SUM(monto) DESC) AS posicion
FROM pedidos
GROUP BY cliente_id;
Índices filtrados/parciales: Reducen tamaño y mejoran la selectividad al aplicarse solo a un subconjunto de datos (ej. solo pedidos activos).
